# Contributing to System Design Master Guide

Thank you for your interest in contributing to the **System Design Master Guide**!  
This is a beginner-friendly, interview-focused open-source repository. Every contribution — big or small — helps the community learn better.

---

## 📌 Project Overview

**System Design Master Guide** is a structured knowledge base for learning how large-scale systems are designed.

**Purpose of this repository:**
- Build a high-quality, open reference for system design learning
- Explain core concepts in simple, practical language
- Help contributors learn by writing and improving technical content

**Who this is for:**
- Students learning distributed systems
- Developers improving architecture knowledge
- Engineers preparing for system design interviews
- Anyone who wants clear, real-world system design learning material

---

## 🍴 How to Fork the Repository

1. Open the original repository on GitHub.
2. Click **Fork** (top-right) to create a copy under your account.
3. Open your fork and copy its URL.
4. Clone your fork locally:

```bash
git clone https://github.com/<your-username>/fuzzy-engine.git
cd fuzzy-engine
```

5. Create a feature branch:

```bash
git checkout -b feature-branch
```

6. Push your branch when ready:

```bash
git push origin feature-branch
```

---

## 🔄 Contribution Workflow

1. Create a new branch for your change.
2. Add or improve content in relevant files.
3. Review your edits for clarity and correctness.
4. Commit with a meaningful message.
5. Push the branch to your fork.
6. Open a Pull Request (PR) from your forked branch to the main repository.

**Example commit command:**

```bash
git add .
git commit -m "docs: improve scalability section examples"
```

---

## ✅ Contribution Guidelines

Please follow these rules before submitting:

- Keep content clean, organized, and easy to navigate.
- Do not add duplicate, off-topic, or low-value content.
- Follow the existing formatting and heading style.
- Ensure all system design concepts are technically accurate.
- Write explanations in beginner-friendly language.

---

## 💡 What You Can Contribute

You are welcome to contribute in these areas:

| Area | Examples |
|------|---------|
| System Design Concepts | Load balancing, caching, sharding, CAP theorem |
| Architecture Diagrams | Diagram explanations and walkthroughs |
| Real-World Case Studies | How Netflix, Twitter, Uber scale their systems |
| Interview Questions | Common system design interview Q&A |
| Performance & Reliability | Scalability patterns, fault tolerance |
| Documentation | Fix typos, improve clarity, restructure sections |

---

## 📬 Pull Request Rules

Before opening a PR, make sure you:

- [ ] Use a clear, specific PR title
- [ ] Add a description explaining what changed and why
- [ ] Link the related issue (if available)
- [ ] Keep PRs small, focused, and easy to review
- [ ] Follow the formatting and style of existing content

---

## 🤝 Code of Conduct

- Be respectful and professional in all discussions.
- Do not submit spam, promotional, or irrelevant changes.
- Support a collaborative, inclusive learning environment.
- Constructive feedback is always welcome.

---

## 🆘 Getting Help

If you need support:

- Open a GitHub **Issue** for bugs, mistakes, or content requests.
- Ask clear questions in issues with context and file references.
- Be patient and constructive when collaborating with maintainers and contributors.
