## 🧠 1. What is a “Service” in MNC systems?

In large companies, an application is split into **services**—independent components that handle specific jobs (login, payments, search, etc.).

This avoids one huge backend and makes systems scalable.

---

# 🏗️ 2. Types of Service Architectures (used in MNCs)

## 1. Monolith Architecture

👉 Everything in one codebase

### Used when:

* Small apps
* Early startup stage
* Simple CRUD systems

### Example usage:

* Internal tools
* Small SaaS MVPs

### Pros:

* Easy to build
* Easy deployment

### Cons:

* Hard to scale
* One bug can crash whole system
* Slow updates

---

## 2. Microservices Architecture (Most used in MNCs)

👉 System split into small independent services

Used by:
Amazon
Netflix
Uber

---

### 🧩 Example microservices:

* User Service (login/register)
* Order Service
* Payment Service
* Notification Service
* Product Service

---

### 🧠 Why MNCs use it:

✔ Independent scaling (payment scales separately)
✔ Independent deployment
✔ Fault isolation (one service crash ≠ full system crash)
✔ Easy to use different tech stacks

---

### ⚠️ Problems:

* Complex architecture
* Network communication overhead
* Debugging is hard

---

## 3. SOA (Service-Oriented Architecture)

👉 Older version of microservices

Used in:

* Banks
* Government systems
* Large legacy enterprises

---

### Difference from microservices:

* Bigger services
* Centralized communication (ESB)

---

## 4. Serverless Architecture

Used by:
Amazon Web Services
Google Cloud

---

### Used when:

* Event-driven apps
* APIs with unpredictable traffic
* Startup scaling quickly

### Example:

* Image processing
* Notification triggers
* Short-lived APIs

---

### Pros:

✔ No server management
✔ Auto scaling
✔ Pay per use

### Cons:

* Cold start delay
* Vendor lock-in

---

## 5. Event-Driven Architecture

Used by:
Netflix
Uber

---

### How it works:

Instead of direct calls:

> Service A → emits event → Service B reacts

---

### Example:

Order placed → event sent →

* Payment service processes
* Inventory updates
* Email sent

---

### Tools used:

Apache Kafka

---

# 🧠 3. When to use what (VERY IMPORTANT)

## 🟢 Monolith:

* Small project
* MVP stage
* Low traffic system

---

## 🔵 Microservices:

* Large systems (MNC scale)
* Millions of users
* Need independent scaling

---

## 🟡 Serverless:

* Lightweight APIs
* Event triggers
* Fast scaling startup apps

---

## 🟣 Event-driven:

* Real-time systems
* High concurrency
* Async workflows

---

# 🏢 4. Real MNC Usage Pattern (VERY IMPORTANT)

Top 50 MNCs NEVER use only one type.

They use **hybrid architecture**:

### Example (Amazon-like system):

* Microservices → core system
* Event-driven → order flow
* Serverless → image processing
* Monolith → legacy admin panels

---

# ⚙️ 5. Why Microservices dominate MNCs

## 🔥 Key reasons:

### 1. Scalability

Only heavy services scale (not whole system)

### 2. Team independence

Each team owns one service

### 3. Faster deployment

No need to deploy full system

### 4. Fault isolation

Payment crash ≠ website crash

# ⚠️ 7. Common Mistake Beginners Make

❌ Using microservices for small projects
❌ Over-engineering simple apps
❌ Not handling network latency
❌ No monitoring/logging system
